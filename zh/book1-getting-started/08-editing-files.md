# 第八章：改文件

## Claude 是怎么修改你的文件的？

当你让 Claude Code 帮你修改代码时，它用的不是"全部重写"的方式，而是一种叫做**查找-替换**的精确机制。

这就像一位外科医生做手术：不是把你整个器官拿出来换掉，而是精确地找到需要处理的那个地方，做最小范围的修改。

**实际上发生了什么：**

1. Claude 先读取你的文件，完整理解现有内容
2. 定位需要修改的具体位置（几行代码、一个函数、一个配置项）
3. 生成修改方案：删除旧的内容，插入新的内容
4. 向你展示"before vs after"的对比
5. 等你确认后，才真正写入文件

这种方式的好处是：**你可以在修改落地之前看清楚改了什么**，如果不对可以立刻拒绝。

---

## 接受前审查更改

Claude Code 在修改文件前，会展示一个**差异对比（diff）**，类似 Git 的 diff 格式：

```
准备修改 src/utils/validator.js：

-  function validateEmail(email) {
-    return email.includes('@');
-  }

+  function validateEmail(email) {
+    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
+    return emailRegex.test(email);
+  }

- 表示删除的行（红色）
+ 表示新增的行（绿色）

是否接受这个修改？ [y/n]
```

**如何判断是否接受：**

- 检查修改的范围是否和你预期一致（没有动到不该动的地方）
- 理解新代码的逻辑（可以直接问 Claude："你改的这段代码是什么意思？"）
- 如果不确定，可以输入 `n` 并提出疑问

**在 VS Code 插件里：** 差异会以可视化的左右对比形式展示，更直观，你可以逐行查看，甚至手动接受部分修改。

## 如何描述你想要的修改

提出修改请求时，越具体越好。

### 描述方式一：说明变化

```
把 validateEmail 函数改得严格一点，用正则表达式来验证格式，
而不是只检查是否包含 @ 符号。
```

### 描述方式二：指明位置

```
src/config.js 第 15 行，把超时时间从 3000 改为 5000。
```

### 描述方式三：描述目标行为

```
当用户输入为空时，表单提交按钮应该变成灰色、不可点击状态。
帮我实现这个效果。
```

### 描述方式四：提供示例

```
我希望错误信息显示成这样：
"用户名不能为空，请重新输入"

而不是现在这样：
"Error: empty username"

帮我修改一下提示文字。
```

## 多文件编辑

一个功能往往涉及多个文件。Claude Code 可以在一个任务中同时修改多个文件。

**例子：添加一个新的 API 接口**

```
帮我添加一个 /api/user/profile 接口，用来获取用户资料。

需要：
1. 在 routes/api.js 里添加路由定义
2. 在 controllers/UserController.js 里添加处理函数
3. 在 models/User.js 里确认有 getProfile 方法（如果没有就添加）
```

Claude 会依次处理这三个文件，每个文件修改前都会展示 diff，让你确认。

**建议：** 多文件编辑时，保持 Git 的工作状态是干净的（先 commit 已有的修改）。这样即使某个文件改错了，也容易追踪和回滚。

## 改错了怎么撤销？

这是新手最担心的问题。答案是：**有很多种方式可以撤销。**

### 方法一：在确认前拒绝

这是最简单的保护。在 Claude 展示 diff 后，你输入 `n` 表示拒绝，修改就不会发生。

### 方法二：使用 Git 撤销

如果你的项目在用 Git（应该用！），可以随时撤销最近的修改：

```bash
# 查看哪些文件被修改了
git status

# 放弃对某个文件的修改（回到上次提交的状态）
git checkout src/utils/validator.js

# 放弃所有未提交的修改（谨慎使用！）
git checkout .
```

### 方法三：使用检查点（Checkpoints）

Claude Code 支持**检查点**功能（在 VS Code 插件中更明显）。在对话中，你可以把鼠标悬停在之前的某条消息上，选择"回滚到这里"——代码会恢复到那个时间点的状态。

### 方法四：让 Claude 帮你撤销

如果刚改完就发现不对，可以直接说：

```
刚才那个修改有问题，帮我改回去。
```

或者更具体：

```
validate 函数改坏了，帮我把它改回只检查 @ 符号的简单版本。
```

### 方法五：手动编辑

如果修改范围很小，直接打开文件手动改回去也很快。

## 编辑请求的最佳实践

### 一次只做一件事

**不推荐：**
```
帮我重构整个 utils 目录，顺便把命名规范统一一下，
把注释翻译成英文，然后把所有函数提取成独立文件。
```

**推荐：**
```
先帮我把 utils/validator.js 里的函数命名从下划线风格
改成驼峰命名风格（比如 validate_email → validateEmail）。
其他的等这步完成后再说。
```

一次一个小改动，每次都能看清楚变化，出了问题也好追踪。

### 说清楚"为什么"，而不只是"怎么做"

**不够好：**
```
把 fetchUser 函数里的 try-catch 去掉。
```

**更好：**
```
fetchUser 函数里的错误处理不符合我们项目的规范——我们
统一在上层处理错误，不在每个函数里 try-catch。
帮我把这个函数的 try-catch 去掉，让错误自然向上抛。
```

告诉 Claude "为什么"，它能做出更符合你意图的修改，避免以后同样的问题再出现。

### 先问后改（对不熟悉的代码）

如果你不太确定某段代码的影响范围，先问清楚再让 Claude 改：

```
我想把 config.js 里的 timeout 值从 3000 改成 10000，
这个改动会影响哪些地方？有什么潜在风险吗？
```

得到解释后再决定是否修改。

### 保持 Git 良好状态

每次 Claude 帮你完成一个完整功能后，及时提交：

```
帮我把刚才的修改都提交到 Git，写一个合适的 commit message。
```

这样你始终有一个可以回退的安全点。

## 一个完整的编辑工作流

以下是一个修改现有功能的典型流程：

**场景：** 你的网站表单，邮箱验证太宽松了，要改得严格一点

```
步骤一：先了解现状
你：帮我看一下 src/utils/validator.js，
    里面的邮箱验证函数现在是怎么写的？

Claude：（读取文件，解释当前逻辑）
        目前 validateEmail 只检查字符串是否包含 '@' 符号，
        没有验证格式是否完整。

步骤二：提出修改需求
你：帮我改成用正则表达式验证，
    要能检测出 "user@domain.com" 这种完整格式。

Claude：（展示 diff）
        准备修改 validateEmail 函数...
        [显示 before/after 对比]
        是否接受？

步骤三：审查并确认
你：（看完 diff，确认逻辑正确）y

步骤四：测试
你：帮我用几个测试用例验证改动是否正确，
    包括：正常邮箱、缺少@的字符串、@后面没有域名的情况。

步骤五：提交
你：改得没问题，帮我提交这个修改。
```

---

## 小结

- Claude 用**查找-替换机制**修改文件，精确而非重写
- 每次修改前都有 **diff 预览**，让你在落地前审查
- 提出修改请求时，**具体** 比 **模糊** 效果好很多
- **多文件编辑** 完全可以，但建议一步步来
- 出错了不用怕：Git、检查点、或者直接让 Claude 改回去
