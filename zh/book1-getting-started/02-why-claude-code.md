# 第二章：为什么选 Claude Code？

## 工具箱里已经有这么多 AI 了，为什么还需要它？

这是一个合理的问题。现在 AI 工具铺天盖地：ChatGPT、Copilot、Cursor、Gemini……你已经在用一个或者几个了。那么 Claude Code 到底解决了什么不同的问题？

简单回答：**当你需要 AI 真正"动手"帮你处理项目——而不只是"出主意"——的时候，Claude Code 是最直接的选择之一。**

让我们用几个真实场景来说明。

---

## 场景一："我接手了一个完全看不懂的项目"

小明刚入职，技术负责人丢给他一个 GitHub 仓库说："你来维护这个，有问题问代码。"打开文件夹，里面有 200 个文件，零文档，注释全是英文，部分注释还是三年前写的。

**用网页版 Claude：** 你把某一个文件复制粘贴进去，Claude 解释了那个文件。然后你又复制另一个文件……你需要一遍遍地搬运代码，Claude 永远只看到你给它看的那一片拼图，很难给你整体的理解。

**用 Claude Code：** 你在那个项目文件夹里打开终端，输入 `claude`，然后说：

```
这个项目是做什么的？帮我解释一下整体结构。
```

Claude Code 会自己去翻阅所有文件，给你一个全局概览。你再追问：

```
认证逻辑在哪里？帮我解释登录流程。
```

它会找到相关文件，把登录流程从头到尾讲给你听，指出哪些文件负责哪个环节。这是网页版无法做到的——因为它能**主动探索**，而不是被动等你喂信息。

---

## 场景二："我想做个网站，但我不会编程"

小红有一个想法：做一个简单的个人简历网站。她不会写 HTML，不会写 CSS，不知道从哪里开始。

**用网页版 Claude：** Claude 会给她一段 HTML 代码，她复制下来，不知道放在哪里，不知道怎么打开……她需要反复问"这段代码放在哪个文件里？""怎么在浏览器里看到效果？"

**用 Claude Code：** 小红在桌面创建一个空文件夹，在终端进入那个文件夹，然后对 Claude Code 说：

```
帮我创建一个简单的个人简历网站，包含我的名字、自我介绍、和联系方式。
```

Claude Code 会：
1. 创建 HTML 文件
2. 写好样式（CSS）
3. 把内容填进去
4. 问你是否要在浏览器里预览

整个过程，小红不需要知道"文件放在哪"——Claude Code 直接帮她创建了。她需要做的只是说清楚自己想要什么，然后确认每一步。

---

## 场景三："我需要自动化一个重复的任务"

小李每周都要做同一件事：从 20 个 Excel 文件里提取数据，汇总成一个报告。这件事手动做需要两个小时，而且容易出错。

他听说可以写个脚本自动化，但他不会编程。

**用 Claude Code：** 小李把那些 Excel 文件放在一个文件夹里，然后告诉 Claude Code：

```
我这里有一批 Excel 文件，每个文件里有一张叫"销售数据"的表格。
帮我写一个脚本，把所有文件里第 B 列（销售额）加总，
输出成一个汇总表格。
```

Claude Code 会：
1. 看一眼你的 Excel 文件，了解实际结构
2. 写一个 Python 脚本
3. 运行它（如果你同意）
4. 给你看结果

如果结果不对，小李可以直接说"汇总数字不对，应该从第 2 行开始，第 1 行是标题"——Claude Code 会修改脚本、重新运行。

---

## 什么时候用 Claude Code，什么时候用其他工具？

工具没有好坏，只有合不合适。这里是一个简单的参考指南：

### 适合用 Claude Code 的情况

- 你需要 AI **读取和修改** 你电脑上的实际文件
- 你要处理一个有**多个文件**的项目，需要整体理解
- 你想让 AI **直接执行命令**（运行程序、安装包、运行测试）
- 你有重复性的**自动化任务**需要脚本来完成
- 你接手了一个**不熟悉的代码库**，需要快速理解
- 你要管理 **Git 操作**（提交、分支、查看改动）

### 适合用网页版 Claude / ChatGPT 的情况

- 你只是想**聊聊想法**、头脑风暴
- 你有一段代码想让 AI **解释或审查**，手动粘贴进去就够了
- 你需要**写文章**、生成文字内容，不涉及文件操作
- 你没有在自己电脑前，用手机或平板访问

### 适合用 GitHub Copilot / Cursor 的情况

- 你主要想要**边写代码边补全**（在编辑器里实时提示）
- 你的工作模式是在 IDE（编辑器）里写代码，想要 AI 辅助
- 你更喜欢图形界面操作，不习惯终端

> **小提示：** 这些工具不是互斥的。很多人同时用 Claude Code 来管理项目、用 Cursor 来写代码、偶尔用网页版 Claude 来讨论方案。找到适合自己的组合就好。

---

## 什么时候不该用 Claude Code？

这一点很重要，很多教程不会告诉你，但我们要直说：

### 不要让 Claude Code 操作你不理解后果的生产系统

"生产系统"是指真正在运行、真实用户在使用的系统。如果你让 Claude Code 在你公司的数据库上跑命令，而你不确定那个命令是否安全——请先弄清楚再动手。

Claude Code 是强大的，这也意味着它做错事的影响会很直接。**永远不要跳过它的确认提示。**

### 不要在没有版本控制的情况下让它大规模修改文件

如果你没有用 Git（版本控制工具）备份你的文件，Claude Code 改了之后后悔就没法轻易找回原版。好消息是 Claude Code 每次改文件前会做快照，但养成用 Git 的习惯是更好的保险。

### 不要期待它永远正确

Claude Code 有时会犯错——写出有 bug 的代码，误解你的需求，或者走一条不必要的弯路。它是一个极好的助手，但你仍然是项目的负责人。看它做的每一步，理解它在干什么，才是最安全的使用方式。

---

## 生产力倍增器：真正的价值在哪里？

最后，让我们说一个宏观视角的问题：为什么值得花时间学习 Claude Code？

**答案是：它改变了你能启动项目的门槛。**

以前，"我想做个工具/网站/脚本"和"这个工具真的运行起来了"之间，隔着漫长的学习曲线——你需要学 HTML，学 Python，学命令行，踩无数坑，才能看到第一个能用的东西。

现在，你可以**先让 Claude Code 帮你做出一个能跑的版本，然后再慢慢理解它是怎么回事**。这个顺序的改变，让很多人从"想法永远只是想法"变成了"想法两小时内能变成原型"。

这不是说你不需要学习——长远来看，理解你在做什么非常重要。但 Claude Code 让你可以**边做边学**，而不是必须**先学才能做**。
