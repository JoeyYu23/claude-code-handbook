# 第六章：权限模式

## Claude Code 为什么要征求你的许可？

想象一下，你雇了一位装修工人来翻修房间。有两种工人：

- **第一种**：每次动工前都向你确认，"我准备把这面墙刷成白色，可以吗？"
- **第二种**：拿到钥匙直接开干，等你回来已经面目全非。

Claude Code 默认是第一种。**它不会在你不知情的情况下悄悄改动你的文件或运行命令。**

但同时，Claude Code 也知道，有时候你已经很熟悉了，不想每次都确认。所以它提供了三种权限模式，让你根据情况自己选择。

---

## 三种权限模式

### 模式一：默认模式（Default Mode）

**特点：** 每次有实质性操作前，Claude 都会征求你的同意。

这是最安全的模式，也是新手最适合的模式。每当 Claude 要：
- 创建或修改文件
- 运行命令（如 `npm install`、`git commit`）
- 删除文件
- 访问敏感资源

……它都会先停下来，告诉你它打算做什么，然后问你："可以吗？"

**示例对话：**

```
Claude: 我准备修改 src/app.js 文件，将第 42 行的变量名从
        `data` 改为 `userData`，以便和其他地方保持一致。

        是否允许此操作？ [y/n]

你:     y
```

**什么时候用：**
- 你刚开始学习 Claude Code
- 处理重要项目或不熟悉的代码库
- 你想理解 Claude 每一步在做什么

---

### 模式二：自动接受模式（Auto-Accept Mode）

**特点：** Claude 可以不经确认，自动完成文件编辑。

这个模式有时被称为 "accept-edits" 模式。Claude 可以直接修改文件，无需你每次按 `y` 确认。但注意：运行 shell 命令仍然需要你确认（除非进入完全的 yolo 模式）。

**什么时候用：**
- 你已经熟悉 Claude 的工作方式
- 在进行大量重复性的代码修改
- 你信任当前的任务，不想被频繁打断
- 做有版本控制保护的项目（Git 随时可以回滚）

**注意：** 自动接受不等于无限制。Claude 仍然有内置的安全边界——它不会删除你整个项目，也不会运行危险命令。

---

### 模式三：计划模式（Plan Mode）

**特点：** Claude 先制定完整方案，等你审批后再执行。

这是一种"先思考，后行动"的工作方式。Claude 会先：
1. 分析你的需求
2. 列出完整的执行计划（修改哪些文件、运行什么命令、逻辑顺序是什么）
3. 等你审批

只有你确认计划没问题，Claude 才开始真正动手。

**示例：**

```
你:     帮我给这个项目添加用户登录功能

Claude: 我来制定一个实现计划：

        【计划概览】
        1. 创建 src/auth/login.js — 登录逻辑
        2. 创建 src/auth/session.js — 会话管理
        3. 修改 src/routes/index.js — 添加登录路由
        4. 修改 src/app.js — 引入认证中间件
        5. 创建 public/login.html — 登录页面

        预计修改 5 个文件，新建 3 个文件。

        是否批准这个计划，开始执行？ [y/n/修改]
```

你可以在这里提问、要求调整，或者直接批准。

**什么时候用：**
- 做大型功能开发，想先看全局
- 不确定 Claude 会怎么实现某个功能
- 向别人演示时，想展示 AI 的决策过程
- 代码审查场景，需要在动手前先了解方案

## 如何切换模式

### 方法一：在对话中按快捷键

在 Claude Code 的交互界面里，按 **`Shift + Tab`** 可以在三种模式之间循环切换。

每次切换，界面底部会显示当前模式。

### 方法二：使用 `/config` 命令

在 Claude Code 里输入：

```
/config
```

这会打开设置界面，你可以在里面设置默认的权限模式。

### 方法三：VS Code 插件设置

如果你在 VS Code 里使用 Claude Code，可以在设置里找到 `Claude Code: Initial Permission Mode`，设置默认启动的模式。

选项说明：
- `default` — 默认模式（推荐新手）
- `plan` — 计划模式
- `acceptEdits` — 自动接受文件编辑
- `bypassPermissions` — 跳过所有确认（危险！慎用）

### 方法四：在提示里直接说明

有时你只是临时想改变方式，可以直接告诉 Claude：

```
先帮我列出修改计划，不要直接动手。
```

```
接下来几步你直接改，不用问我了。
```

## 允许列表和拒绝列表

除了三种模式，Claude Code 还支持更精细的权限控制：**允许哪些操作，禁止哪些操作**。

这在 `.claude/settings.json` 文件里配置：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Read(~/.zshrc)"
    ],
    "deny": [
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./secrets/**)"
    ]
  }
}
```

**解读：**
- `allow` 列表里的操作，Claude 无需询问就能执行
- `deny` 列表里的操作，Claude 无论如何都不能执行

**实际例子：**

你可以允许 Claude 自动跑测试（因为这很安全），但禁止它访问 `.env` 文件（里面有 API 密钥）：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(npm run build)"
    ],
    "deny": [
      "Read(./.env)",
      "Bash(rm -rf *)"
    ]
  }
}
```

## 安全理念：为什么 Claude 要问你？

很多 AI 工具的思路是"越自动化越好"。Claude Code 的设计理念有所不同：**人始终是决策者，AI 是执行者。**

这背后有几个原因：

**1. 你的代码很重要**
代码库是你劳动的结晶。一个错误的操作可能毁掉几小时的工作。让你确认每一步，是对你工作的尊重。

**2. AI 可能理解有偏差**
Claude 很聪明，但不是万能的。它对你需求的理解可能和你的真实意图有出入。审批流程让你有机会在"出错之前"纠正。

**3. 学习机会**
当 Claude 告诉你它打算做什么时，这本身就是一次学习：你看到了问题的解法、命令的用法、代码的组织方式。直接帮你做完，你反而什么都没学到。

**4. 边界明确**
明确的权限边界让 Claude 不会"越权"——它永远不会在你睡觉时悄悄修改你的生产代码。

---

## 实际建议：什么时候用哪种模式？

| 场景 | 推荐模式 | 理由 |
|------|----------|------|
| 学习 Claude Code 的前两周 | 默认模式 | 了解每个操作，建立直觉 |
| 探索陌生代码库 | 计划模式 | 先看全局，再深入 |
| 大型功能开发 | 计划模式 | 防止方向偏差 |
| 熟悉的小改动 | 自动接受 | 提升效率 |
| 重构大量代码 | 自动接受 + Git 保护 | 效率优先，靠版本控制兜底 |
| 生产环境代码 | 默认模式 | 谨慎永远没错 |

---

## 小结

- **默认模式**：Claude 每步都问，最安全，适合新手和重要项目
- **自动接受模式**：文件编辑无需确认，适合熟练用户
- **计划模式**：先出方案再动手，适合大任务和审查场景
- **允许/拒绝列表**：精细控制哪些操作可以/不可以执行

Claude Code 的权限系统不是障碍，而是保护你的工具。你越了解它，就能越放心地让 Claude 帮你工作。
