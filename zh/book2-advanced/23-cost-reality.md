# 第二十三章：费用真相 — Claude Code 到底花多少钱

每个人都在问、但没人愿意诚实回答的问题：**Claude Code 用起来到底要花多少钱？**

官网定价页只展示套餐价格，博客文章只介绍功能特性，却没人公开自己的真实用量数据。本章来做这件事。以下所有数字均来自作者本人七周的实际使用记录——一名同时推进多个并行项目、重度使用 Claude Code 的开发者。

---

## 理解定价模式

数据之前，先梳理一下各方案的选项：

**Pro 套餐（$20/月 ≈ ¥145/月）**
对 Claude 模型的限速访问。适合偶尔使用 Claude Code 的开发者——每周用几个小时。触发速率限制时需要等待。不会产生超额费用，没有意外账单。

**Max 5 套餐（$100/月 ≈ ¥725/月）**
比 Pro 更高的速率限制。面向工作日中持续使用 Claude Code、但不运行大规模并行工作流的开发者。

**Max 20 套餐（$200/月 ≈ ¥1,450/月）**
最高订阅档位。专为将 Claude Code 作为主要开发工具的开发者设计——重度使用、多个项目、并行智能体。下文的用量数据均来自此套餐。

**API 直接调用（无基础费用 + 按 token 计费）**
无需订阅，纯用量付费。每一个 token 都要花钱。适合 CI/CD 流水线、自动化工作流，或那些使用频率低且波动较大、订阅套餐反而浪费的场景。务必做好预算管理——无上限的使用量可能迅速烧钱。

**Team/Enterprise（按席位计价）**
Team 套餐起步价 $25/用户/月（按年计费，≈ ¥180/用户/月）。Enterprise 价格需要谈判。两者均包含管理员控制、审计日志和额外安全特性。一旦团队统一采用 Claude Code，这个方案就值得认真考虑。

### 读懂这组数据的规律

每周数据的波动幅度很大，值得深究：

**W1–W2（$147、$243）：** 热身阶段。配置工作流、建立使用习惯、较短的 session。

**W3–W4（$621、$705）：** 进入全速状态。多个项目同时推进，并行智能体运行，处理复杂任务的长时 session。

**W5 峰值（$1,065）：** 最贵的一周。高强度开发：多个并行智能体同时运行，大量使用 Opus 模型处理复杂推理任务，session 长时间运行未执行 compact。在 API 计费下这一周本会超过 $1,000，订阅套餐下无需额外付一分钱。

**W6 骤降（$211）：** 最可能的解释：触及速率限制。经历极高强度的 W5 之后，使用量被节流。这是订阅套餐的典型规律——有上限，一旦触及，吞吐量就会下降。

**W7（3天内 $419）：** 月底恢复高强度使用。

核心洞察：**使用量是突发性的，不是线性的。** 有的周用量很轻，有的周极为密集。按 token 计费模式会让密集那几周的费用高得令人望而却步。订阅套餐吸收了这种波动。

## Max 套餐的杠杆计算

这笔账值得算清楚，因为数字确实惊人。

按 API 费率（2026年初近似值）：
- Claude Sonnet：约 $3/百万输入 token，约 $15/百万输出 token
- Claude Opus：约 $15/百万输入 token，约 $75/百万输出 token

高强度开发周——大文件反复读取、长时代码生成 session、开启扩展思考、多个并行智能体——轻松消耗数千万 token。W5 达到 $1,065 的 API 等价费用，对于这类使用强度来说并不罕见。

Max 20 套餐无论用量多少，固定收费 $200/月（≈ ¥1,450/月）。也就是说：

| Month | API-Equivalent Used | Plan Cost | Leverage |
|-------|-------------------|-----------|----------|
| February | $991 | $200 | 5.0x |
| March (25 days) | $2,421 | $200 | 12.1x |

3月份，每花 1 美元，换回了价值 12 美元的算力。对于原本需要按 API 计费的开发者而言，Max 20 套餐的盈亏平衡点大约在月均 $200（≈ ¥1,450）的 API 等价用量。超过这个量之后，性价比会越来越高。

**结论很直接：对重度用户而言，Max 20 相对 API 直接计费明显定价偏低。** Anthropic 实际上是在通过订阅套餐补贴重度使用者。随着方案迭代，这种情况未必会永远持续，但目前就是这个现实。

## 什么操作烧 Token 最快

并非所有任务的 token 消耗都相同。以下是来自真实经验中消耗最快的几类场景，按消耗速率排序：

**1. 读取大文件**
读取一个 2,000 行的文件，每次访问就会向上下文注入数千个 token。如果 Claude 在一个 session 内多次读取同一个大文件——在跨文件调试时这很自然——token 会快速累积。一个 500 行的文件读三遍，等同于读一次 1,500 行的文件。

**2. 使用 Opus 模型**
Opus 每个 token 的费用大约是 Sonnet 的 5 倍。同一个任务在 Sonnet 上用 10 万 token，换算到 Opus 上就是 50 万 token 等价。能力差距确实存在，但把 Opus 反射性地用于 Sonnet 完全能胜任的任务，是最大的不必要成本来源。

**3. 长时 session 不执行 `/compact`**
Claude Code 的上下文随着每次交互不断增长。每条新消息都包含完整的对话历史。一个开始了一个小时、经历了 50 轮交互的 session，每条新消息都在发送此前全部 50 轮的上下文。这既昂贵，又往往没必要——那些早期上下文大多早已不再相关。

**4. 并行子智能体**
每个子智能体从零开始、拥有独立的上下文窗口。如果你派生 5 个智能体并行工作，就意味着同时支付 5 个独立上下文窗口的费用。这通常是值得的权衡——并行性带来的收益超过了成本——但必须明白：并行智能体是在倍增 token 消耗，而不是共享它。

**哪些操作不会大量消耗 token：**
- 简短、聚焦的问题，回复也短
- Haiku 模型任务（日志智能体、简单查询、结构化数据转换）
- 提示词缓存命中——CLAUDE.md 等频繁复用的内容会被缓存，费用约为正常输入 token 价格的 10%
- 保持聚焦、干净退出的短时 session

## 真正有效的降本策略

以下是在不显著损失能力的前提下降低成本的实用方法：

**简单任务使用 `/effort low`**
需要快速查询、简单重构或基础问答时，告诉 Claude 这是低强度任务。这会传达出不需要深度推理的信号，限制不必要的计算。

**主动使用 `/compact`**
不要等上下文膨胀到很大再处理。在子任务切换时就执行 `/compact`——调试完一个 bug 之后、实现完一个功能之后、开始新事项之前。一份紧凑的已完成工作摘要，远比携带完整对话历史便宜。

```
# 在长 session 内切换任务时
/compact

# 针对特定内容压缩
/compact focus on the API schema changes we made
```

**将 Sonnet 作为默认模型**
Sonnet 能以高质量完成绝大多数真实开发任务——功能实现、bug 修复、代码审查、重构、测试编写。将 Opus 留给真正需要深度推理的场景：复杂的架构决策、微妙的多系统调试、研究综合。一个简单的经验法则：如果你能为这个任务写出清晰的规格说明，Sonnet 大概率就够了。

**子智能体做简单工作时使用 Haiku**
不是每个智能体都需要完整的 Sonnet 能力。如果一个智能体只是在做日志记录、简单查询、格式转换，或按照明确规则处理结构化数据，Haiku 速度快、费用低，完全够用。

```bash
# 以编程方式派生智能体时指定模型
claude --model haiku -p "Append this log entry to the journal file: ..."
```

**自动化场景使用 `--bare` 模式**
在脚本或 CI 流水线中非交互式运行 Claude Code 时，`--bare` 模式跳过交互式 UI 的加载开销。启动更快，浪费的 token 更少。

```bash
claude --bare -p "Run the test suite and report failures"
```

**写一份聚焦的 CLAUDE.md**
CLAUDE.md 的每一行都会在每次 session 开始时加载到上下文。一份包含过时笔记、冗余说明和组织脚手架的 500 行 CLAUDE.md，每次 session 的成本远高于一份精炼的 150 行版本。定期审计你的 CLAUDE.md，删除不再活跃使用的内容。

**用 `.claudeignore` 排除大型目录**
如果项目中包含 Claude 永远不需要读取的目录——构建产物、依赖目录、大型数据文件、生成资产——明确排除它们。Claude 读不到它不知道的东西，但如果它在探索项目结构时意外涉及大型目录，就会把这些目录都读进来。

```
# .claudeignore
node_modules/
dist/
build/
data/raw/
*.parquet
*.csv
```

## 该选哪个套餐？

一个实用的决策框架：

**Pro（$20/月 ≈ ¥145/月）**
你偶尔使用 Claude Code 寻求帮助——提问、审查代码、起草文档。每周少于 5 小时。速率限制虽然烦人，但不会造成阻塞。在摸清自己的使用量之前，这是正确的起点。

**Max 5（$100/月 ≈ ¥725/月）**
你在工作日中持续使用 Claude Code，但主要是聚焦型任务。每周 5–20 小时。你频繁触及 Pro 的限制，想摆脱这种挫败感。不运行并行智能体或重量级自动化工作流。

**Max 20（$200/月 ≈ ¥1,450/月）**
Claude Code 是你的主要开发工具。每周 20 小时以上，多个活跃项目，并行智能体工作流，session 动辄运行数小时。到了这个档位，杠杆计算才真正令人信服。根据上文数据，相对 API 直接计费的盈亏平衡点大约在月均 $200（≈ ¥1,450）的 API 等价用量——而重度用户通常在月初第一周就会超过这个量。

**API 直接调用（按量计费）**
你在构建一个以编程方式调用 Claude Code 的产品。你有 CI/CD 流水线。你的使用量是突发性的、可预测的，不值得订阅。或者你希望对每个 token 都有成本可见性和控制权。务必做好预算——在 Anthropic 控制台设置硬性上限。

**Team/Enterprise（按席位计费）**
团队中有不止一人在定期使用 Claude Code。你需要管理员控制、用量可见性或合规特性。Team 套餐（$25/用户/月，按年计费，≈ ¥180/用户/月）一旦每个席位的使用强度达到中等水平，就具备经济性。

## 监控你的用量

不要靠猜——靠量。以下工具帮你获得可见性：

**会话内费用追踪：**
```
/cost
```
显示当前 session 的 token 用量和费用。随时执行，随时掌握进度。

**套餐用量与限额：**
```
/usage
```
显示当前用量相对套餐限额的位置。适合判断距离速率上限还有多远。

**用 ccusage 查看历史数据：**
```bash
# 当月每日明细
npx ccusage@latest daily

# 按模型分类的每周视图
npx ccusage@latest weekly --breakdown

# 会话级详情
npx ccusage@latest sessions
```

`ccusage` 读取本地 Claude Code 日志并生成用量报告。不需要 API key——完全依赖本地数据。安装一次，纳入每周复盘流程。

**要盯住的核心指标：** 每周 API 等价费用。如果你在 Max 20 套餐上，每周 API 等价费用超过 $200（≈ ¥1,450），说明你获得了出色的杠杆比。如果只有 $50/周，你可能选对了套餐，也可能该降到 Max 5。数据会告诉你答案。

## 诚实的总结

如果按 token 付费并重度使用，Claude Code 是贵的。对于一个运行并行智能体、多个项目、动辄数小时 session 的严肃开发者，按 API 费率的真实成本是每月数百到数千美元。

Max 20 订阅套餐按数字来看，对重度用户而言是一笔异常划算的交易——至少截至 2026 年初是这样。在 $200/月（≈ ¥1,450/月）固定费用上获得 8 倍以上的杠杆，意味着你可以毫无顾忌地高强度使用 Claude Code，而不必因为担心费用就克制自己。这改变了工作时的心理状态：任务确实需要更强模型时你会毫不犹豫地选 Opus，并行智能体有帮助时你就开，session 需要跑多久就跑多久。

这种杠杆比是否会随着 Anthropic 调整定价而持续，尚不可知。趁它还在，用好它。
