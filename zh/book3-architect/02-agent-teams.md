# 第 2 章：智能体团队

## 什么是智能体团队？

想象用单个 Claude Code 智能体构建产品 — 你要求它写后端，它做。然后你要求它写前端，同时它在写前端，后端闲置等待你的下一条指令。

现在想象你可以*有多个 Claude Code 智能体并行工作*。一个写后端。另一个写测试。第三个审查代码。它们通过共享任务列表协调、相互递交工作，并向你报告结果。这是智能体团队。

一个**团队**包括：

- **一个主导智能体**（编排者）— 这个智能体与你交互并协调团队的工作。
- **多个队友智能体**（工作者）— 这些智能体声称任务、执行它们，并将完成报告回共享任务列表。
- **一个共享任务列表** — 所有智能体看到相同的要完成的工作列表。
- **智能体间消息** — 智能体可以相互提问并共享上下文。

结果是工作发生得更快，因为智能体不被阻止等待你给指令 — 它们不断拉取任务并并行执行。

---

## 何时使用团队 vs. 子智能体 vs. 单独

在深入如何设置团队之前，让我们清楚*何时*你应该首先使用团队。这很重要：不是每个项目都受益于团队。

### 何时使用单个智能体：

- **工作本质上是顺序的。** 你在调试需要深入上下文的问题。每一步都会通知下一个。
- **项目很小**（1–3 个文件，<2 小时工作）。团队协调开销超过节省时间。
- **你需要紧密反馈循环。** 你探索设计空间并快速迭代。

### 何时使用子智能体：

- **工作平行但松散耦合。** 需要两个智能体在不同关切上工作，但不需要共享任务列表。
- **你想保持简单。** 子智能体更容易设置。
- **工作自然分解成 2–3 块。**

### 何时使用智能体团队：

- **多个智能体需要在共享任务列表上工作。** 有大的积压（20+ 任务）并受益于连续平行执行。
- **工作需要频繁递交。** 任务有依赖并以增量方式发现。
- **项目很大且复杂。** 9+ 文件、多个子系统、估计 4+ 小时工作。
- **你重视可观察性。** 团队提供清晰视图：谁在处理什么、什么被阻止、什么完成。

## 启用智能体团队

智能体团队是 Claude Code 中的实验功能。要使用：

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
claude
```

或添加到你的 shell 配置文件：

```bash
echo 'export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1' >> ~/.zshrc
```

启用后，通过开始对话启动团队：

```
I want to use agent teams for this project. Can you set up a team with a lead and 3 teammates?
```

## 团队架构

### 主导智能体

主导是你的主要接触：

- **从你接受任务** — 用自然语言描述高级工作。
- **创建结构化任务** — 分解请求为离散任务，添加到共享任务列表。
- **协调团队** — 检查进度、重新分配被阻止的任务、上升问题。
- **向你报告** — 总结完成了什么。

主导不执行实现工作 — 仅编排。

### 队友智能体

队友是工作者：

- **声称任务** — 当任务可用且依赖满足时声称。
- **执行任务** — 读文件、写代码、运行测试、调试。
- **报告完成** — 标记任务完成并包括总结。
- **提问** — 用 `SendMessage` 询问澄清。

经验法则：对大多数项目目标 2–4 个队友。

### 共享任务列表

所有智能体看到相同的任务列表。任务有状态：
- `open` — 未声明
- `in_progress` — 队友在工作
- `blocked` — 无法继续
- `completed` — 完成

## 显示模式

### 自动（默认）

Claude Code 选择最好的显示模式。通常显示状态行：

```
[Lead] ✓ Ready to accept tasks
[Teammate-01] idle
[Teammate-02] idle
```

### 进程中

所有智能体在同一终端。看实时工作流。

### TMUX

打开 TMUX 会话，每个智能体一个窗格：

```bash
export CLAUDE_CODE_DISPLAY_MODE=tmux
```

### ITERM2

在单独的 iTerm2 选项卡中打开：

```bash
export CLAUDE_CODE_DISPLAY_MODE=iterm2
```
